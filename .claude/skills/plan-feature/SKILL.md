# plan-feature

새 기능을 **계획서로 먼저 합의하고**, **GitHub 이슈로 추적 가능하게 만든 뒤** 코딩을 시작하기 위한 워크플로우 스킬. CLAUDE.md §1·§1-B 의 "한 Issue = 한 커밋, 작은 단위로 쪼갠다" 규약과 SPEC/USER_JOURNEY 단일 출처 원칙을 그대로 따른다.

## 언제 사용

수강생이 다음 같은 의도를 표현할 때:

- "기능 G 간트 뷰 계획 세워줘"
- "다음 기능 이슈 만들어줘"
- "오버듀 기능 어떻게 구현할지 plan 파일로 정리해줘"
- "feature F 작업 시작 준비해줘"

`/plan-feature G` 처럼 슬래시 호출도 가능.

## 선행 조건

호출 즉시 다음을 **병렬 Bash 한 번**으로 점검한다. 하나라도 실패하면 그 단계의 보완을 안내하고 멈춘다.

- `test -f SPEC.md && test -f USER_JOURNEY.md` — 두 문서가 저장소 루트에 있는가
- `test -d plan || mkdir -p plan` — `plan/` 디렉터리 준비 (없으면 생성)
- `gh auth status` — GitHub CLI 로그인 상태
- `git remote get-url origin` — **origin** 의 URL (수강생 본인 저장소). 이슈는 **반드시 origin** 에 등록하므로 이 값이 정답.
- `gh repo set-default $(git remote get-url origin | sed -E 's#.*[:/]([^/]+/[^/]+)\.git#\1#')` — `gh` 의 default repo 를 origin 으로 고정. fork 저장소에서는 default 가 upstream 으로 잡혀 있어 이 단계를 건너뛰면 이슈가 upstream(예: 강사 저장소)에 잘못 등록된다.
- `git status --porcelain` — 작업 트리가 깨끗한지 (필수는 아니나 안내 목적)

## 절차

### Phase 1 — 대상 기능 식별

1. 수강생이 어떤 기능을 만들지 명시했는지 확인. 모호하면 **AskUserQuestion** 또는 직접 질문으로 1회 좁힌다 (예: "SPEC.md 어느 섹션의 기능인가요? §6 기능 F인가요?").
2. `SPEC.md`에서 해당 섹션을, `USER_JOURNEY.md`에서 대응 시나리오 ID(J{n})를 찾아 **한국어 한두 줄로 요약** 후 수강생에게 확인 받는다.
3. 충돌 처리:
   - SPEC에 정의가 **없으면** → "먼저 `SPEC.md`에 항목을 추가할까요, 아니면 이번 계획에서 함께 추가할까요?" 묻고 합의.
   - SPEC `§9 범위 밖`에 있는 기능이면 → "범위 밖 항목입니다. 범위 안으로 옮기려면 `SPEC.md`부터 수정해야 합니다. 계속할까요?" 확인.

### Phase 2 — 코드베이스 탐색 (필요 시)

기능이 새 모듈을 다수 건드릴 것 같거나 기존 패턴 재사용을 확인해야 하면 **Explore 에이전트 1개**(또는 최대 3개 병렬)를 띄워 다음을 수집한다:

- 진입점/툴바/관련 화면 파일 경로
- 재사용 가능한 Server Action·헬퍼·컴포넌트
- 기존 테스트 컨벤션(헬퍼, fixture 위치)

탐색이 자명하면 건너뛴다. 추측보다 한 번의 탐색이 낫지만, 단순 기능에는 과한 도구 사용을 피한다.

### Phase 3 — 계획서 초안 작성

`plan/feature-{slug}.md` 를 작성한다. **slug 명명 규칙:** kebab-case, 기능 영역을 짧게. 기존 예시: `feature-d-delete.md`, `feature-f-csv.md`. 형식: `feature-{문자/번호}-{핵심키워드}.md`.

**의무 섹션 구조 (기존 plan 파일 컨벤션):**

```
0. Context        — 왜 이 변경을 하는가, 무엇이 비어 있는가, 끝나면 어떤 시나리오/체크리스트가 통과해야 하는가
1. 범위           — 포함/제외 (SPEC §9 범위 밖 항목과 명확히 구분)
2. 아키텍처 결정  — 표 형식. 결정 / 근거. 한 줄씩.
3. 파일별 변경    — 신규/수정 파일과 핵심 시그니처·구조
4. 수정 금지      — 손대지 않을 파일과 그 이유 (회귀 방지)
5. 구현 순서      — "한 커밋 = 한 논리 단위" 단계. 각 단계의 검증 방법 포함.
6. 검증           — E2E/타입/린트/수동 회귀 항목. USER_JOURNEY ID로 지칭.
7. 위험·보류 항목 — SPEC 미정 결정과 그 폴백 선택 사유
8. 참고 파일 인덱스 — 표 형식. 경로 / 역할.
```

작성 시 지켜야 할 점:
- 모든 SPEC 인용은 **섹션 번호로** (예: "SPEC §6 F-2"). 추측 없이 인용.
- USER_JOURNEY 시나리오는 **ID로** 지칭 (예: "J11", "J12").
- 코드 스니펫은 시그니처 수준까지만. 본 구현 코드를 plan에 다 옮기지 않는다.
- **plan 파일은 plan 모드의 산출물이 아니라 일반 마크다운 파일**이다. Write 툴로 직접 만든다.

초안이 끝나면 수강생에게 핵심 요약(2~4줄)과 함께 "이대로 이슈 등록할까요? 수정할 부분이 있으면 알려주세요"라고 묻는다. 수정 요청이 있으면 Edit 으로 반영하고 다시 묻는다. **승인 전에는 다음 단계로 넘어가지 않는다.**

### Phase 4 — GitHub 이슈 등록

수강생 승인 후 `gh issue create` 로 등록한다.

> ⚠️ **반드시 origin 저장소에 등록한다.** upstream(강사/원본) 저장소는 본인이 권한이 없거나, 다른 수강생들의 학습 흐름을 어지럽힌다. 선행 조건의 `gh repo set-default` 가 origin 을 가리키는지 다시 확인하고, 의심되면 `gh issue create --repo $(git remote get-url origin | sed -E 's#.*[:/]([^/]+/[^/]+)\.git#\1#') ...` 처럼 `--repo` 로 명시한다.

**이슈 제목 컨벤션** — `기능 {레터} — {짧은 제목}` (필요 시 `(Sub-feature)`):
- 예: `기능 F-1 — CSV 내보내기`
- 예: `기능 G — 간트 일정 시각화`

**이슈 본문** — HEREDOC 으로 다음을 전달:

```markdown
## 배경
{SPEC.md §X 인용 + 한두 줄 요약}

## 사용자 여정
- {USER_JOURNEY ID + 한 줄 요약}

## 계획 문서
`plan/feature-{slug}.md` 에 상세 계획. 본 이슈는 그 계획의 추적 단위.

## 작업 체크리스트
- [ ] {plan §5 구현 순서의 단계 1}
- [ ] {plan §5 구현 순서의 단계 2}
- ...
- [ ] E2E: {대응 J{n} 시나리오들}
- [ ] 타입·린트·전체 E2E 회귀

## 완료 조건
- [ ] {plan §6 검증의 핵심 항목}
- [ ] PR 머지 후 `main` 의 db-migrate 워크플로우(스키마 변경 시) 성공
```

명령 예시:
```bash
gh issue create \
  --title "기능 F-1 — CSV 내보내기" \
  --body "$(cat <<'EOF'
## 배경
... (위 템플릿 채움)
EOF
)"
```

**라벨/마일스톤은 기본값.** 수강생이 라벨 체계를 가지고 있으면 그때 `--label feature` 등을 추가. 본 스킬은 임의로 라벨을 만들지 않는다.

이슈 생성에 성공하면 `gh issue create` 의 출력에서 이슈 URL 과 번호를 추출해 수강생에게 보여준다.

### Phase 5 — 마무리 안내

```
✅ 계획서 + 이슈 준비 완료
- 계획: plan/feature-{slug}.md
- 이슈: #{번호} {URL}

다음으로 보통 다음을 합니다 (이 스킬 범위 밖):
1) 새 브랜치 생성·체크아웃 — 예: git checkout -b feat/{slug}
2) plan §5 구현 순서대로 코딩 시작
3) 각 단계 끝에 npm run lint && npx tsc --noEmit && npm run test:e2e

이어서 코딩을 시작할까요?
```

수강생이 "네"라고 하면 이 스킬은 종료하고 일반 코딩 흐름으로 돌아간다. 브랜치 자동 생성·자동 코딩은 본 스킬이 하지 않는다.

## 컨벤션 참고 (이미 있는 모범 사례)

저장소에 이미 존재하는 두 plan 파일이 황금 표준이다 — 새 plan을 작성할 때 구조와 디테일 깊이를 이 둘과 맞춘다:

- `plan/feature-d-delete.md` — 작은 표면(액션 1건 + 다이얼로그 1건). 0~8 섹션의 좋은 최소치.
- `plan/feature-f-csv.md` — 큰 표면(라이브러리 도입 + 2-step 다이얼로그 + 헬퍼 추출). 0~10 섹션까지 확장 가능.

`plan/` 디렉터리 자체에는 README 가 없다 — 디렉터리 본질이 "기능별 계획"이라 자명. 별도 README 만들지 않는다.

## 규칙

- **이슈는 항상 origin 저장소에 등록한다.** upstream(예: 강사 원본 저장소)에 등록하지 않는다. fork 환경에서는 `gh` 의 default repo 가 upstream 으로 잡히기 쉬우므로 매 호출 직전 `gh repo set-default` 또는 `--repo` 명시로 origin 을 강제한다.
- **한국어로 대화한다.** 이 저장소는 수강생용이다.
- **각 단계는 한 번에 하나씩 승인 받는다** — 계획 작성 후 한 번, 이슈 등록 전 한 번. 자동으로 묶어 진행하지 않는다.
- **이슈 본문은 plan 파일을 가리키되, plan 본문을 통째로 복붙하지 않는다.** 변경이 생기면 plan 파일 한 곳만 갱신해도 추적되도록.
- **SPEC.md / USER_JOURNEY.md 를 마음대로 수정하지 않는다.** 두 문서를 바꿔야 할 일이 생기면 별도 PR로 제안하고, plan 파일에는 "후속 PR 로 SPEC §X 갱신"이라고 명시.
- **`gh issue create` 가 실패하면** (예: gh 미인증, 네트워크) 계획 파일은 이미 만들어져 있으므로 그대로 두고, 수강생에게 인증·네트워크 점검 안내 후 재시도. 계획 파일을 지우지 않는다.
- 이 스킬은 **브랜치 생성·체크아웃·푸시·코딩**을 하지 않는다. 마무리 안내까지만.
